# Feature Specification: 任务更新流程优化

**Feature Branch**: `001-update-tasks`  
**Created**: 2025-11-21  
**Status**: Draft  
**Input**: User description: "更新任务"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时更新单个任务 (Priority: P1)

项目经理需要在工作看板中快速调整任务标题、描述、优先级与截止日期，以响应突发需求变化，同时希望立即同步到所有协作者。

**Why this priority**: 单任务更新是最常见、最关键的操作，对交付节奏影响最大，也是任何任务系统的基础能力。

**Independent Test**: 在测试环境中，选定一条任务记录，修改多个字段并保存，确认界面刷新后的数据与 API 返回值一致，且通知相关成员。

**Acceptance Scenarios**:

1. **Given** 用户拥有编辑权限，**When** 在详情面板中同时修改任务标题与截止日期，**Then** 系统在 2 秒内保存并显示最新信息。
2. **Given** 任务被他人锁定编辑，**When** 用户尝试更新，**Then** 系统提示“存在更高版本”并展示差异，要求用户刷新后再提交。

---

### User Story 2 - 批量状态与负责人更新 (Priority: P2)

运营团队每周需要根据冲刺计划，对多条任务批量调整状态与负责人，避免逐条修改导致的效率低下。

**Why this priority**: 批量操作可显著减少重复劳动，提高计划变更时的吞吐量，在单任务更新稳定后推进。

**Independent Test**: 勾选多条任务，统一修改状态与负责人，验证所有任务都获得相同的更新记录且失败项有明确回滚提示。

**Acceptance Scenarios**:

1. **Given** 用户勾选 10 条任务，**When** 批量设置状态为“进行中”，**Then** 成功任务的状态同时更新，并生成单次批处理日志。
2. **Given** 批处理中包含无权限编辑的任务，**When** 提交批量更新，**Then** 系统跳过无权限项并在结果摘要中列出失败原因。

---

### User Story 3 - 更新历史与通知 (Priority: P3)

合规负责人需要追溯任务曾被谁、在何时修改过哪些字段，并在关键字段变化时触发订阅通知，确保审计可追踪。

**Why this priority**: 历史追踪与通知增强透明度与治理能力，在单/批更新功能可用后提供价值。

**Independent Test**: 对同一任务进行多次更新，查看历史时间线与通知记录，确认字段差异被记录且通知发送给订阅人。

**Acceptance Scenarios**:

1. **Given** 任务开启关键字段通知，**When** 优先级被调整为“紧急”，**Then** 系统向关注者推送包含修改人、时间与新值的消息。
2. **Given** 审计人员打开任务历史，**When** 过滤“截止日期变动”事件，**Then** 列表准确展示所有相关版本与差异摘要。

---

### Edge Cases

- 并发更新：在用户提交更新时检测版本号，若已被他人修改则返回冲突提示并提供最新快照对比。
- 不合法字段：若输入超过字段上限或格式错误（如日期格式），阻止保存并在对应字段显示校验信息。
- 批处理部分失败：批量更新过程中若部分任务失败，需要在事务结束后提供成功/失败清单，并保证成功项不会被回滚。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须允许具备编辑权限的用户在任务详情面板中修改标题、描述、优先级、负责人、截止日期以及自定义字段。
- **FR-002**: 系统必须在提交前对必填字段与数据格式进行客户端 + 服务端双重校验，失败时阻止写入并返回字段级错误信息。
- **FR-003**: 系统必须采用乐观并发控制（版本号或 ETag），当检测到版本不一致时返回冲突错误并附带最新任务快照。
- **FR-004**: 系统必须在单次保存中支持部分字段更新，未变动字段不得被覆盖，确保差分写入。
- **FR-005**: 批量更新操作必须支持按筛选结果或手动勾选清单执行，对每个任务独立验证权限。
- **FR-006**: 批量更新必须以幂等请求提交，后台在 30 秒超时内提供进度与失败原因，失败任务不得影响已成功的其它任务。
- **FR-007**: 系统必须记录每一次更新的操作者、时间、字段前后差异，并存储至少 12 个月供审计查询。
- **FR-008**: 系统必须允许用户为关键字段（状态、负责人、截止日期、优先级）订阅通知，满足触发条件时在 1 分钟内发送。
- **FR-009**: 所有更新操作必须写入事件流或消息队列，供搜索索引与报表同步使用，若下游不可用需重试并记录告警。
- **FR-010**: API 与界面都必须遵循相同的权限模型，未获编辑权限的用户只能查看更新历史而无法写入。

### Key Entities *(include if feature involves data)*

- **Task**：表示单个工作项，包含基础元数据（标题、描述、状态、优先级、截止日期、负责人、自定义字段、版本号）。
- **TaskChangeLog**：记录一次任务变更，字段包括任务 ID、变更人、变更时间、字段差异（old/new）、触发来源（单次/批量/API）。
- **BulkUpdateJob**：代表一次批量更新请求，字段包含任务集合、批次 ID、执行状态、成功/失败统计、每个失败项原因列表。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 单任务更新从点击保存到后端确认的 95th 响应时间 ≤ 2 秒。
- **SC-002**: 批量更新 100 条任务的整体成功率 ≥ 98%，失败项均包含明确原因并在 5 秒内返回给用户。
- **SC-003**: 关键字段通知在触发后 1 分钟内送达订阅用户的比例 ≥ 99%。
- **SC-004**: 任务历史查询可在 2 秒内返回最近 50 条变更记录，且审计抽查合格率 100%。
